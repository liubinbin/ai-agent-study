# 工具执行由用户决定

在日常使用 AI 的过程中，会发现 AI 在执行时会征询我们的意见，让我们选择同意或不同意。接下来看看这个功能是如何实现的。

## 初始化

QwenPaw 中的 QwenPawAgent 继承了 ToolGuardMixin 和 ReActAgent，是处理用户请求的实际执行者。询问功能由 ToolGuardMixin 插件实现。

## 何时触发询问

ToolGuardMixin 重写了 `_acting` 和 `_reasoning` 方法，在 ReActAgent 推理和执行之前先处理 ToolGuardMixin 的逻辑。其中 `_reasoning` 无任何特殊逻辑，询问逻辑是在 `_acting` 中实现的。

因此 ToolGuardMixin 是在 agent 实际执行 act 之前处理询问的。

## _acting 方法

1. 确保 ToolGuardMixin 完成初始化。
2. 决定是否需要启动 guard 操作类型。处理场景包括两个：
   - 一、根据配置实现不需要同意、完全拒绝某些 tool 等场景；
   - 二、工具执行层级包括 strict、smart、auto、off 几种，同样根据配置。
3. guard 操作类型有 needs_approval、auto_denied 和空三种。空则调用 ReActAgent 执行；若为 auto_denied，则调用 `_acting_auto_denied`；若为 needs_approval，则调用 `_acting_with_approval`。

## _acting_auto_denied 方法

拼装拒绝 msg，返回给用户，并加入 memory。

## _acting_with_approval 方法

1. 先从 ApprovalService 获取一个 PendingApproval 并返回给用户。PendingApproval 中有一个 future，用于等待用户输入，并将所有 PendingApproval 加入 ApprovalService 的缓存中，以 request_id 作为缓存键。
2. 然后开启一个带超时时间的等待线程，线程内就是等待 PendingApproval 里的 future。

## ApprovalService 如何设置

用户提交审批结果是通过 HTTP 请求。从上述逻辑可以看出，要让系统知道审批结果，就是通过 request_id 找到 pending，然后为 pending 的 future 设置内容，完成整个审批流程。

1. FastAPI 中设有 approval 路由，相关的 HTTP 请求实现在 src/qwenpaw/app/routers/approval.py。
2. 其中 `post_approval_approve` 和 `post_approval_deny` 负责处理审批请求，方法内获取 request_id，然后调用 ApprovalService 的 `resolve_request`。
3. ApprovalService 的 `resolve_request` 方法逻辑与本小节开头描述类似。由此可见，ApprovalService 综合处理 agent 发起的审批请求以及用户对审批的输入。
