# 小助理学会了先思考再行动

在 deer-flow 项目中，使用了 LangGraph 作为 agent 实现，下面看一下具体是怎么使用 LangGraph 的。

## 代码层面
创建 agent 代码在 /backend/packages/harness/deerflow/agents/lead_agent/agent.py L342 create_agent 方法创建了一个 agent，传入参数共5个。
1. model：agent 使用的模型
2. tools：提供给 agent 使用 tool
3. middlerware：中间件为在 agent 执行期间，插入的一些流程，可以使整个主流程代码更加清晰
4. system_prompt：系统 prompt，拼接了长期记忆、skill、和一些对执行控制的要求。
5. state_schema：deer-flow 对原 AgentState 做了扩展，增加了部分结构。

## 部署层面
上面的方法通过 langgraph.json 配置文件中的 graphs.lead_agent 指定。
deer-flow 启动一个 langgraph server，在启动时会读取 langgraph.json 文件。
下面是关于涉及到 langgraph 的启动流程，目前只考虑开发模式的启动。
1. 在使用手册中，可以看到开发模型下通过 make docker-start 启动服务
2. 在 makefile 中，make docker-start 实际调用的是 ./scripts/docker.sh start
3. 在 docker.sh 可以看到会启动 frontend gateway langgraph nginx，使用的是 docker/docker-compose-dev.yaml 文件
4. 在 docker-compose-dev.yaml 中可以看到 langgraph 服务，通过 uv run langgraph dev 启动 langgraph 服务

## 其他配置
在 langgraph.json 中，除了 graphs.lead_agent，还可以看到 checkpointer.path 参数，用于定时保存对话信息，以供出问题时恢复。

## 使用层面
在 backend/docs/ARCHITECTURE.md 文件中，我们可以看到 langgraph server 承担 Agent Runtime、Thread Mgmt、SSE Streaming，是作为与前端的交互的主要服务，承担接收请求、处理请求、返回请求等功能。

## langgraph 的 create_agent 方法
这个方法实际上是创建了第一个 react agent，从[1]中可以看到是一个 model 与 tools 的循环，直到任务全部完成，才组装最终结果。有一个这个分解+{反思+执行}，小助理就有了自主接收和处理任务的能力。
