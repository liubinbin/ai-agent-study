# 小助理处理事情开始学会套路

小助理在处理事情的时候，逐渐发现了有些事情是套路，例如要签订合同需要让法务看一下，例如合同签订完需要保留到某处，不能自己存。这些套路就意味着我们在处理某些事情的前后需要多留一个心眼，可能需要多想一下。
对于 deerflow 项目，此类事情是通过插件体系实现。

## LangGraph 中间件框架
根据[1]，我们可以知道 agent 循环就是接收请求，发送至 LLM，根据返回的计划，不停地去调用对应的工具，最终将工具的结果合并输出返回给用户。
从循环中，我们可以看到有几个点是可以暴露出来，下面罗列 LangGraph 暴露的中间件 hook。
1.before_agent：如 ThreadDataMiddleware，功能是在 sandbox 初始化需要工作目录，例如 workspace、uploads、outputs，在调用 agent 之前。
2.before_model：如 ViewImageMiddleware，功能是注入图片信息，在调用 LLM 之前。
3.wrap_model_call：暂无
4.after_model：如 SubagentLimitMiddleware，功能是控制 subagent 的个数，需要在调用 LLM 生成 subagent 计划之后，实际执行之前。
5.wrap_tool_call：如 GuardrailMiddleware，功能是评估是否需要拒绝工具执行，需要在包装 tool 调用之前。
6.after_agent：如 MemoryMiddleware，功能是处理本次完整请求的消息，评估后更新长期记忆，需要在调用 agent 之后。

## deerflow 项目中间件列表
•ThreadDataMiddleware：创建线程目录
•UploadsMiddleware：扫描上传文件
•SandboxMiddleware：获取沙箱
•DanglingToolCallMiddleware：补缺失 ToolMessage
•GuardrailMiddleware：Guardrail 仅支持自定义实例
•ToolErrorHandlingMiddleware：处理 tool 调用失败
•SummarizationMiddleware：上下文压缩
•TodoMiddleware：扩展原 TodoListMiddleware，处理上下文压缩式，丢失部分 write_todos 的情况
•TitleMiddleware：生成标题
•MemoryMiddleware：入队记忆
•ViewImageMiddleware：注入图片 base64
•SubagentLimitMiddleware：截断多余 task
•LoopDetectionMiddleware：检测循环
•ClarificationMiddleware：拦截 ask_clarification
